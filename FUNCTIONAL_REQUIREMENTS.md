# DataBuddy AI - Functional Requirements Specification

## Overview
DataBuddy AI is a Conversational Database Studio (IDE) that allows users to interact with a PostgreSQL database using natural language. The system translates prompts into executable SQL, runs commands safely, and visualizes database schema and tables in real-time within a 3-panel workspace.

---

## 1. Core Functional Requirements

### 1.1 AI Chatbot (The Command Center)
**Specification:**
- **Purpose:** Primary interface for user interaction with natural language
- **Location:** Bottom-right panel (50% height of right side)
- **Functionality:**
  - Accept natural language prompts from users
  - Display user messages in chat history
  - Show loading state ("Generating SQL...") during translation
  - **DO NOT display system messages** - only user prompts should appear
  - Auto-scroll to latest message
  - Support multi-line input (Shift+Enter for new line)
  - Auto-resize textarea based on content (max 120px height)

**Expected Behavior:**
- User types: "Create a table called Employees with name, email, salary"
- Chatbot shows only the user's message
- SQL is generated and executed automatically in background
- No system confirmation messages appear in chatbot

---

### 1.2 SQL Execution & Live Progress (The Engine)
**Specification:**
- **Purpose:** Display generated SQL and execution progress
- **Location:** Top-right panel (50% height of right side)
- **Functionality:**
  - Display SQL generated by AI translation
  - Show real-time execution logs from backend
  - Display execution progress: `Connecting... -> Executing -> Success: X rows affected`
  - Show error messages if execution fails
  - Color-code logs (success/error/warning)
  - Auto-scroll to latest log entry

**Expected Behavior:**
- After user prompt, SQL appears in SQL Panel
- Terminal section shows: `[✓] Connecting to database...`
- Then: `[✓] Executing CREATE TABLE statement...`
- Then: `[✓] Success: Table created successfully`
- If error: `[❌] Error: Table already exists`

---

### 1.3 Real-Time DB Visualization (The Source of Truth)
**Specification:**
- **Purpose:** Display database schema and table data
- **Location:** Left panel (70% width, full height)
- **Functionality:**
  - **Schema Browser:**
    - Table selector dropdown (populated from database schema)
    - Auto-select first table on startup if tables exist
    - Auto-select newly created table after CREATE TABLE
    - Refresh button to reload schema
  - **Data Grid:**
    - Display table columns and rows
    - Pagination (50 rows per page default)
    - Search/filter functionality
    - Row count display
    - Empty state: "No table selected or no data available"
    - Loading state during data fetch
  - **Auto-Refresh:**
    - Automatically refresh schema after CREATE/ALTER/DROP operations
    - Automatically refresh table data after INSERT/UPDATE/DELETE operations
    - Show updated data immediately after operations

**Expected Behavior:**
- On startup: Fetch schema, auto-select first table, display data
- After CREATE TABLE: Schema refreshes, new table appears in dropdown, auto-selected
- After INSERT: Table data refreshes showing new rows
- User can manually select different tables from dropdown
- User can navigate pages using pagination controls

## 2. Natural Language to SQL Translation

### 2.1 Translation Engine
**Specification:**
- **Backend:** Uses Groq AI (Llama 3.3 70B) for translation
- **Input:** Natural language prompt + optional schema context
- **Output:** PostgreSQL-compatible SQL query
- **Auto-Execution:** SQL executes automatically after generation (unless conversational)

**Supported Operations:**
1. **CREATE TABLE**
   - User: "Create a table called Employees with name, email, salary"
   - SQL: `CREATE TABLE employees (id SERIAL PRIMARY KEY, name VARCHAR(255), email VARCHAR(255), salary DECIMAL(10,2))`
   - Auto-adds `id SERIAL PRIMARY KEY` column
   - Infers appropriate data types

2. **ALTER TABLE (Add Columns)**
   - User: "Add phone number column to employees table"
   - SQL: `ALTER TABLE employees ADD COLUMN phone_number VARCHAR(255)`
   - Checks existing columns to avoid duplicates
   - Only adds columns that don't exist

3. **INSERT**
   - User: "Add John with salary 50000 to employees"
   - SQL: `INSERT INTO employees (name, salary) VALUES ('John', 50000)`
   - Never includes `id` column (auto-generated)

4. **UPDATE**
   - User: "Change John's salary to 60000"
   - SQL: `UPDATE employees SET salary = 60000 WHERE name = 'John'`
   - Always includes WHERE clause

5. **DELETE**
   - User: "Remove John from employees"
   - SQL: `DELETE FROM employees WHERE name = 'John'`
   - Requires WHERE clause

6. **TRUNCATE**
   - User: "Remove all data from employees table"
   - SQL: `TRUNCATE TABLE employees`
   - Used for clearing all data (faster than DELETE)

7. **SELECT**
   - User: "Show me all employees"
   - SQL: `SELECT * FROM employees LIMIT 50`
   - Auto-adds LIMIT 50 for safety

8. **DROP TABLE**
   - User: "Delete the employees table"
   - SQL: `DROP TABLE employees`
   - Requires confirmation for destructive operations

**Conversational Handling:**
- Greetings ("hello", "hi") → Returns `NOT_A_DATABASE_QUERY`
- Farewells ("thanks", "bye") → Returns `NOT_A_DATABASE_QUERY`
- Non-database queries → Returns `NOT_A_DATABASE_QUERY`
- No system messages shown in chatbot for these cases

## 3. SQL Execution

### 3.1 Execution Flow
**Specification:**
1. **Validation:**
   - Validate SQL syntax
   - Check for forbidden operations
   - Verify SQL safety constraints

2. **Destructive Operation Confirmation:**
   - DROP TABLE → Show confirmation dialog
   - TRUNCATE TABLE → Show confirmation dialog
   - DELETE without WHERE → Show confirmation dialog
   - User must confirm before execution

3. **Execution:**
   - Connect to PostgreSQL database
   - Execute SQL query
   - Handle timeouts (default 10 seconds)
   - Support query cancellation

4. **Result Handling:**
   - Return rows and columns for SELECT queries
   - Return rows_affected for INSERT/UPDATE/DELETE
   - Return execution logs
   - Handle errors gracefully

5. **Post-Execution:**
   - Refresh schema if DDL operation (CREATE/ALTER/DROP)
   - Refresh table data if DML operation (INSERT/UPDATE/DELETE)
   - Update query history
   - Record performance metrics

### 3.2 Multi-Statement Execution
**Specification:**
- Support multiple SQL statements separated by semicolons
- User: "Create products table and add sample data"
- SQL: `CREATE TABLE products (...); INSERT INTO products VALUES (...);`
- Execute statements sequentially
- Support transactions (optional, via `use_transaction` parameter)
- Rollback on error if using transactions

### 3.3 Query Types Supported
- ✅ SELECT (read data)
- ✅ CREATE TABLE (create tables)
- ✅ ALTER TABLE (modify schema)
- ✅ INSERT (add data)
- ✅ UPDATE (modify data)
- ✅ DELETE (remove specific data)
- ✅ TRUNCATE TABLE (clear all data)
- ✅ DROP TABLE (delete tables)

## 4. Schema Management

### 4.1 Schema Discovery
**Specification:**
- **Endpoint:** `GET /api/schema`
- **Functionality:**
  - Introspect PostgreSQL database schema
  - Return table names and column information
  - Include column types and primary key information
  - Cache schema information
  - Auto-refresh after DDL operations

**Schema Response Format:**
```json
{
  "tables": [
    {
      "name": "employees",
      "description": null,
      "columns": [
        {
          "name": "id",
          "type": "integer",
          "is_primary": true
        },
        {
          "name": "name",
          "type": "character varying",
          "is_primary": false
        }
      ]
    }
  ],
  "last_sync": "2026-03-13T10:00:00Z"
}
```

### 4.2 Schema Refresh
**Specification:**
- Automatically refresh after CREATE TABLE
- Automatically refresh after ALTER TABLE
- Automatically refresh after DROP TABLE
- Manual refresh via button click
- Show loading state during refresh

## 6. Security & Safety

### 6.1 SQL Validation
**Specification:**
- Validate SQL syntax before execution
- Prevent dangerous operations (DROP DATABASE, etc.)
- Check for SQL injection attempts
- Validate query structure

### 6.2 Rate Limiting
**Specification:**
- IP-based rate limiting
- Default: 10 requests per 60 seconds
- Configurable via environment variables
- Persistent storage (survives server restarts)
- Return 429 status when limit exceeded
- Show toast notification to user

### 6.3 Query Timeouts
**Specification:**
- Default timeout: 10 seconds
- Configurable via `QUERY_TIMEOUT` environment variable
- Cancel long-running queries
- Return timeout error to user

### 6.4 SELECT Query Limits
**Specification:**
- Auto-add LIMIT 50 to SELECT queries
- Prevent accidental large data fetches
- Configurable via `SELECT_LIMIT` environment variable
- User can manually specify higher limits if needed

### 6.5 Destructive Operation Confirmation
**Specification:**
- Show confirmation dialog for:
  - DROP TABLE
  - TRUNCATE TABLE
  - DELETE without WHERE clause
- User must explicitly confirm
- Cancel option available

## 7. Error Handling

### 7.1 Error Types
**Specification:**
- **Translation Errors:** AI fails to generate SQL
- **Validation Errors:** SQL fails validation
- **Execution Errors:** Database execution fails
- **Timeout Errors:** Query exceeds timeout
- **Rate Limit Errors:** Too many requests

### 7.2 Error Display
**Specification:**
- Show errors in execution logs (SQL Panel)
- Show toast notifications for critical errors
- **DO NOT show errors in chatbot** (only user messages)
- Provide helpful error messages
- Include suggestions for fixing errors

### 7.3 Error Recovery
**Specification:**
- Auto-fix common issues:
  - Duplicate column errors
  - SERIAL sequence sync issues
- Retry logic for transient failures
- Graceful degradation (continue even if non-critical features fail)

## 12. Auto-Execution Flow

### 12.1 Complete Flow
**Specification:**
1. User enters prompt in Chatbot
2. Frontend calls `POST /api/translate`
3. Backend generates SQL using AI
4. Frontend receives SQL response
5. **Frontend automatically calls `POST /api/execute`**
6. Backend executes SQL
7. Frontend receives execution results
8. Frontend refreshes schema if needed
9. Frontend refreshes table data if needed
10. Frontend auto-selects newly created table

**Key Points:**
- SQL executes automatically (no manual "Run" button needed)
- User only sees their prompt in chatbot (no system messages)
- SQL appears in SQL Panel
- Execution logs appear in SQL Panel
- Results appear in Database Preview
